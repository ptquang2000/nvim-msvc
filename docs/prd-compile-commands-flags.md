# PRD — compile_commands flag correctness: --merge-defaults and --vc-tools-install-dir

*Generated from grill session on 2026-06-21*

---

## Problem Statement

`compile_commands.json` generated by the plugin produces incomplete IntelliSense entries for
projects that use per-file `PrecompiledHeader` overrides, and silently uses the wrong MSVC
toolchain on machines with multiple Visual Studio versions installed.

1. **Missing inherited flags on PCH-override files.** Projects like `WAOnDemand-v12.vcxproj`
   declare per-file `<PrecompiledHeader>NotUsing|Use|Create</PrecompiledHeader>` overrides.
   MSBuild splits these into separate `GetClCommandLines` batches; entries in non-default
   batches omit all `ItemDefinitionGroup`-inherited flags (include paths, preprocessor
   defines, language standard). Without `--merge-defaults`, those entries arrive in
   `compile_commands.json` stripped of their inherited flags, making IntelliSense blind to
   most of the project's include tree.

2. **VCToolsInstallDir mismatch on multi-VS machines.** The plugin passes `--vs-path` derived
   from a version-constrained vswhere query (respecting the user's `vs_version` setting). The
   extractor's internal auto-detection of `VCToolsInstallDir` runs `vswhere -latest`
   unconditionally, ignoring `vs_version`. On a machine with multiple VS installations,
   `--vs-path` correctly points to the version the user selected, while auto-detected
   `VCToolsInstallDir` resolves to whichever VS version happens to be latest on that machine.
   The mismatch causes `SanitizeFallbackPaths` and `VCInstallDir` global property injection to
   reference the wrong toolchain, producing compiler flag paths inconsistent with the
   selected build environment.

---

## Solution

Pass `--merge-defaults` unconditionally in every extractor invocation. Derive
`VCToolsInstallDir` from the already-selected `installationPath` (the same path that becomes
`--vs-path`) by scanning `{installationPath}\VC\Tools\MSVC\*` for the highest-version
subdirectory, then pass it as `--vc-tools-install-dir`. Both changes live entirely in the
plugin; no extractor rebuild is required.

---

## User Stories

1. As a developer working on a project with per-file PCH overrides (`NotUsing`/`Use`/`Create`),
   I want `compile_commands.json` to include all inherited include paths and defines for every
   source file, so that clangd provides accurate IntelliSense for all files regardless of
   their PCH batch.

2. As a developer with multiple VS versions installed who pins `vs_version`, I want
   the compiler flags in `compile_commands.json` to reference the MSVC toolchain that
   matches my selected VS version, so that include paths and toolchain flags are
   consistent with the version I build with.

3. As a developer opening a workspace, I want to trigger `compile_commands.json` generation
   explicitly from the msvc:// buffer action page, so that generation only runs when I
   choose it rather than silently on every solution selection.

4. As a maintainer reading ADR 005, I want the documented behavior of `--deduplicate` to
   include its flag-stripping side effect, so that I understand why `compile_commands.json`
   lacks warning and optimization flags and do not add them back as a "fix."

## Implementation Decisions

### `compile_commands.lua` — `find_vc_tools_install_dir`

New module-local function. Given a VS `installationPath`:
- Appends `\VC\Tools\MSVC` and checks it exists via `Util.is_dir`
- Uses `vim.uv.fs_scandir` to enumerate subdirectories
- Filters to entries whose name parses as a dot-separated version (≥2 numeric parts)
- Returns the path of the highest version by numeric component comparison

Returns `nil` if the MSVC root doesn't exist or contains no valid version directories.
No vswhere call; pure filesystem scan.

### `compile_commands.lua` — `build_argv`

Two additions:
- After `--vs-path`: emit `--vc-tools-install-dir <path>` when `opts.vc_tools_install_dir`
  is non-empty.
- After `--vc-tools-install-dir` (unconditionally): emit `--merge-defaults`.

### `compile_commands.lua` — `M.generate`

Derive `vc_tools_install_dir` once before spawning the extractor pool:
```
vc_tools_install_dir = opts.vc_tools_install_dir
    or (opts.vs_path ~= "" and find_vc_tools_install_dir(opts.vs_path))
```
Pass it to every `build_argv` call in the spawn loop. No change to how `init.lua` passes
`install_path` (the vswhere `installationPath`) as `opts.vs_path`.

### `init.lua` — remove auto-trigger from `set_solution`

Remove the `_run_compile_commands` call from `Msvc:set_solution`. Generation no longer fires
automatically on solution selection.

### `ui.lua` — action page with compile_commands entry

Add a compile_commands action entry to the normal-mode msvc:// buffer. The action page
appears below the separator (after the project list). It shows an actionable line such as
`  [g] Generate compile_commands.json` with the current output path. Pressing the bound key
calls `_run_compile_commands` on the singleton. The entry is only shown when the cc feature
is enabled in config. Status feedback (in-progress, done, error) is shown via the existing
log/build-append mechanism.

### `M._internal` — expose for tests

Add `find_vc_tools_install_dir` to the `M._internal` table so specs can call it directly.

### ADR 005 — new Decision subsections

Three new subsections under "Decision":

**`--merge-defaults` always passed**
Per-file PCH overrides create separate GetClCommandLines batches. Non-default batch entries
omit ItemDefinitionGroup-inherited flags unless backfilled. The extractor's MergeDefaultFlags
is additive (only adds flags not already present) and safe to run unconditionally.
CMake-generated vcxprojs are unaffected (single batch).

**`--deduplicate` strips warning and optimization flags (side effect)**
`CompileCommandDeduplicator.CleanSingleEntry` strips `/W*`, `/WX`, `/wd*`, `/we*`,
`/external:W*`, `/O1`, `/O2`, `/Od`, `/Ob*`, `/GL`, `/Gw` from every entry — including
non-duplicated files. This is a side effect of passing `--deduplicate`, not just
deduplication. The stripped flags are acceptable for IntelliSense use (clangd neither
enforces warnings nor requires optimization flags) but make the output unsuitable for
build reproduction.

**`--vc-tools-install-dir` derived from selected VS installation**
The extractor's auto-detection uses `vswhere -latest` (no version constraint). On machines
with multiple VS versions, this would disagree with the version-constrained `--vs-path`. The
plugin owns VS selection (via `resolve_install` / `vs_version`), so it also owns deriving
VCToolsInstallDir: scan `{installationPath}\VC\Tools\MSVC\*`, pick the highest version
directory, pass as `--vc-tools-install-dir`. This guarantees the two flags always reference
the same installation.

## Testing Decisions

### What makes a good test

- `build_argv` tests: assert the flat `argv` string contains the expected flags in the right
  positions. Cover: `--merge-defaults` present unconditionally; `--vc-tools-install-dir`
  present when provided; `--vc-tools-install-dir` absent when `nil`/empty.
- `find_vc_tools_install_dir` tests: nil/empty input returns nil; non-existent base path
  returns nil. Real filesystem tests (multiple MSVC versions, pick-highest) are skipped
  unless a test fixture directory is set up.

### Modules with tests

- `CC._internal.build_argv` — existing test expanded; three new assertions
- `CC._internal.find_vc_tools_install_dir` — new test block (nil-input guards only)

### Prior art

`compile_commands_spec.lua` already exercises `build_argv` via `table.concat(argv, "|")` and
`s:find(needle, 1, true)` pattern — follow the same style.

## Out of Scope

- Extractor-side fix for `--vs-path` → `VCToolsInstallDir` derivation (rejected: plugin owns
  VS selection; extractor fix would be redundant and require a separate release).
- Passing `--vc-targets-path` from the plugin (same mismatch vector exists but is not a
  known active regression on the verification workload).
- Sentinel entry handling — confirmed acceptable as-is; clangd silently skips non-existent
  files.
- Redundant cmake sub-solution deduplication — confirmed acceptable; `merge_temp_files`
  handles collisions.

## Further Notes

- Verification workload: `grsc-gsdk-win32 / WAOnDemand-v12.vcxproj` with
  `Features\fsd\FSDFeature.cpp` as the per-file PCH canary. After the change,
  `FSDFeature.cpp`'s entry should include the inherited include paths from
  `ItemDefinitionGroup` (previously missing from its non-default batch).
- Machine has multiple VS versions; `grsc-gsdk-win32` is built with VS2022 (the selected
  `vs_version`). The VCToolsInstallDir mismatch is active whenever the machine's latest
  installed VS differs from the user's `vs_version` setting.
- `--deduplicate` ADR note is documentation-only; no code change needed.
