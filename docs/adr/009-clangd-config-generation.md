# ADR 009: .clangd generation alongside compile_commands.json

**Status**: Accepted  
**Date**: 2026-06-21

## Context

`msbuild-extractor-sample` produces `compile_commands.json` with per-file compilation
commands (include paths, defines, flags) for every compiled source unit. This covers `.cpp`
files well. However, clangd has two gaps when working with MSVC projects:

1. **Headers opened in isolation** have no compile_commands entry; clangd falls back to
   built-in defaults and misses project-level preprocessor definitions.
2. **MSVC-only flags** (`/Zc:*`, `/MP`, `/analyze`, `/sdl`, etc.) appear in the extractor
   output and cause clangd to emit parse errors or warnings it cannot act on.

A `.clangd` YAML config at the workspace root can address both: `CompileFlags.Remove` strips
unrecognised flags globally, `CompileFlags.Add` injects project-level defines, and
`CompileFlags.CompilationDatabase` points clangd at the generated JSON.

## Decision

### Generate `.clangd` always alongside `compile_commands.json`

The `g` action in the `msvc://` buffer triggers both. There is no way to generate one
without the other. The `.clangd` file is always overwritten unconditionally; it is treated as
fully plugin-owned.

### Both files written at solution root; `cc.outdir` removed

Both `compile_commands.json` and `.clangd` are written to `Util.dirname(Msvc.solution)`.
The `cc.outdir` config field is removed. The output location is no longer configurable —
it is always the solution directory, which is also where clangd expects `.clangd` to live
(it walks up from the open file looking for `.clangd`).

`cc.builddir` is retained — it still controls where sub-solutions are scanned for the
compile_commands merge step.

### `CompileFlags.Remove` — hardcoded MSVC flag list

A static list of MSVC-only flags is removed unconditionally. No config escape hatch:

```
/Zc:*, /MP, /analyze, /analyze:*, /sdl, /GS, /RTC1, /GL, /Gw, /Gy, /Gm-,
/ZH:SHA_256, /external:*, /Wall
```

These flags either have no clangd equivalent or cause clangd to emit spurious diagnostics.
The list is well-established for MSVC/clangd interop and requires no per-project tuning.

### `CompileFlags.Add` — pinned project defines only, active config/platform

Preprocessor definitions are read from the **pinned project** (`Msvc.project`) only,
using the **active configuration and platform** (`Msvc.settings.configuration` /
`Msvc.settings.platform`) to select the correct `<ItemDefinitionGroup Condition="...">`.

If no project is pinned (`Msvc.project == nil`), `CompileFlags.Add` is omitted entirely.

Rationale: a union of all projects' defines is misleading (conflicting macros both true
simultaneously). Per-project defines already land in `compile_commands.json` for `.cpp` files;
`.clangd` fills the gap for headers, and headers belong to one project context at a time.

### `CompileFlags.CompilationDatabase: .`

Since `.clangd` and `compile_commands.json` are co-located at the solution root, the
database path is always `.` (current directory relative to `.clangd`).

### Parsing `<PreprocessorDefinitions>` per config/platform

A new `discover.parse_vcxproj_defines(vcxproj_path, configuration, platform)` function
reads the vcxproj XML and extracts defines from the matching
`<ItemDefinitionGroup Condition="'$(Configuration)|$(Platform)'=='<cfg>|<plat>'">` block.
Defines are split on `;`, `%(PreprocessorDefinitions)` inheritance tokens are dropped,
and each define is prefixed with `-D` for inclusion in `CompileFlags.Add`.

### `.clangd` YAML structure

```yaml
# Generated by nvim-msvc. Do not edit.
CompileFlags:
  CompilationDatabase: .
  Remove:
    - /Zc:*
    - /MP
    - ...
  Add:           # only present when a project is pinned
    - -DFOO
    - -DBAR=1
```

YAML is written as plain string concatenation — no external library required for this
flat structure.

## Consequences

- `cc.outdir` is removed from the config schema and merge logic. Users who relied on it
  to write `compile_commands.json` elsewhere must move to the solution directory layout.
- `resolve_outpath` in `compile_commands.lua` is simplified: always `Util.dirname(solution)`.
- `discover.lua` gains `parse_vcxproj_defines(path, cfg, plat)` — a new public function.
- `compile_commands.lua` gains `generate_clangd(opts)` — writes the `.clangd` file; called
  from `generate()` after the JSON merge succeeds.
- Headers that include project-specific macros now get correct IntelliSense when the right
  project is pinned in the msvc buffer before generating.
